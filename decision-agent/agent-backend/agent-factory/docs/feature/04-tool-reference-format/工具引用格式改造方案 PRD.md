## 1. 背景与问题定义
在 Decision Agent Platform 中，Dolphin Language 使用 @ 引用来编排工具调用。当前实现依赖工具名称进行引用，当工具名称发生修改时，已有编排内容会出现找不到目标的问题，严重影响编排的稳定性与可维护性。

目前的问题：
+ 工具引用依赖可变的名称，而非稳定的标识符
+ 工具名称变更导致现有编排失效
+ 缺乏类型区分，无法有效支持不同类型的工具（tool/agent/mcp）
+ UI 展示与内部引用耦合，影响用户体验

---

## 2. 目标（Goals）
### 2.1 核心目标
1. **提供稳定的工具引用机制**  
使用 @{tool_type:tool_id} 格式替代 @工具名称，确保引用的稳定性。

2. **支持类型化的工具引用**  
明确区分 tool、agent、mcp 三种工具类型，提供更好的扩展性。

3. **保持用户友好的展示体验**  
UI 层仍显示友好的工具名称，内部使用稳定的 ID 格式。

4. **提供平滑的迁移路径**  
支持新旧格式并存，逐步引导用户迁移到新格式。


---

## 4. 关键概念与约束
### 4.1 工具引用格式
**新格式**：`@{tool_type:tool_id}`

**格式规范**：
- `tool_type`：必须是 tool、agent、mcp 之一
- `tool_id`：对应类型的唯一标识符
- 使用花括号和冒号结构，易于解析和扩展

### 4.2 工具类型定义
| tool_type | 说明 | tool_id 来源 | 示例 |
|-----------|------|--------------|------|
| tool | API 工具 | tool.tool_id | @{tool:123456} |
| agent | Agent 工具 | agent.agent_key | @{agent:my_agent} |
| mcp | MCP 工具 | mcp.mcp_tool_id | @{mcp:search_tool} |

### 4.3 作用域与生命周期
- **引用作用域**：仅在当前 Agent 项目内有效
- **标识稳定性**：tool_id 不会因名称变更而改变
- **向后兼容**：支持老的 @工具名称 格式

### 4.4 UI 展示策略
**内部存储**：使用 @{tool_type:tool_id} 格式
**用户展示**：显示为 "工具名称 (tool_id)"

### 4.5 配置开关
提供 `is_use_tool_id_in_dolphin` 配置项：
- true：启用新格式，创建的引用使用 @{tool_type:tool_id}
- false：使用旧格式，保持向后兼容

---

## 5. 用户故事（User Stories）
### 5.1 创建稳定的工具引用
**作为** Agent 开发者  
**我希望** 使用稳定的 ID 而非名称引用工具  
**以便** 工具名称变更时我的编排不会失效。

**验收标准**：
+ 可以使用 @{tool_type:tool_id} 格式创建引用
+ 修改工具名称后，已有引用仍然有效
+ 支持三种工具类型的引用

---

### 5.2 友好的展示体验
**作为** 业务用户  
**我希望** 在编辑器中看到易懂的工具名称  
**以便** 我能快速理解编排内容。

**验收标准**：
+ UI 显示友好的工具名称而非 ID
+ 可以查看完整的引用信息
+ 支持点击跳转到工具详情

---

### 5.3 平滑的格式迁移
**作为** 系统管理员  
**我希望** 融入老格式  
**以便** 不影响现有业务的使用。

**验收标准**：
+ 支持新旧格式并存
+ 提供自动迁移工具

---

### 5.4 类型化的工具管理
**作为** 平台开发者  
**我希望** 明确区分不同类型的工具  
**以便** 提供更好的扩展性和管理能力。

**验收标准**：
+ 支持按类型过滤和查找工具
+ 新类型可以轻松扩展
+ 类型信息在引用中明确体现

---
