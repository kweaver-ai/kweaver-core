# Task Management

## Overview

### Feature Definition

Ontology index build tasks are a core index management capability provided by the ADP BKN. They support building object indexes for object classes in the **BKN**.  
With this capability, users can perform semantic knowledge queries, overcome the limitations of virtualization-based queries, improve object query efficiency and recall, and quickly locate object instances that meet business conditions.

### Application Scenarios

This feature is suitable for **data analysts** who need to use semantic filtering to locate specific object instances in business analysis scenarios. For example:

- Find enterprises whose names contain `chemical` or related similar concepts
- Filter suppliers whose industry matches `new energy`

The object indexes generated by ontology index build tasks help users quickly obtain lists of objects that meet certain conditions within a target object class, providing support for subsequent analysis such as verifying existence or checking basic information.

## Key Terms

| Term | Meaning |
| --- | --- |
| Full build | A build mode that creates indexes for all data of all object classes |
| Incremental build | A build mode that partially updates changed data with incremental identifiers |
| Task | The top-level unit of index building. A task can include a group of object classes or all object classes |
| Task status | A state used to describe progress, including `Pending`, `Running`, `Completed`, `Failed`, and `Canceled` |

## Prerequisites

Before using task management, make sure the following conditions are met:

- **Permission requirement**: You have access to the target BKN, configured in advance by an administrator.
- **Data readiness**: The object class to be indexed must already be bound to a view. If no bound view exists, the system rejects task creation.
- **Task conflict check**: The object class must not have any running task. If there is one, wait for it to finish or cancel it before creating a new task.

## Operation Guide

### Create an Index Build Task

#### Steps

1. Log in to ADP, open **BKN** in the workspace, and select the target BKN for which you want to create an index task.
2. In the left sidebar, click **Task Management**. On the page, click **Create Task** in the upper-right corner to open the creation dialog.
3. Configure the following parameters:
   - **Task name**: By default, the current BKN name is shown. You can modify it. The maximum length is 40 characters, it cannot be empty, and duplicate names are allowed.
   - **Build mode**:
     - **Full build**: selected by default. Builds indexes for all data of all object classes.
     - **Incremental build**: only updates changed data that has incremental identifiers. If the data source does not define an incremental identifier, the system automatically performs a full build.
4. Click **Confirm** to create the task.

#### Key Notes

- After a task is created successfully, the system triggers it to run automatically in parallel.
- After the whole task succeeds, the system replaces the bound indexes of the object classes in one step. If the task fails, the original indexes are kept and existing query functions are not affected.

### View the Task List

#### Steps

1. Go to the **Task Management** list page and locate the target task.
2. In the actions column, click **View** to see the task details.

### Delete an Index Build Task

#### Steps

1. Open the **Task Management** page and locate the target task.
2. In the actions column, click **Delete**. If the task status is `Running`, the button is disabled.
3. The system shows a confirmation dialog warning that the task and its related details will be deleted and cannot be recovered. Click **Confirm** to complete the deletion.

#### Key Notes

- Only tasks in the `Pending`, `Completed`, `Failed`, or `Canceled` states can be deleted.
- Deleting a task does not affect indexes that have already been built.

## Notes

- **Object class modification limit**: If an object class has a running task, its configuration such as bound view and field properties cannot be modified until the task finishes or is canceled.
- **Task operations**: The current version does not support pausing a running task. To stop a task, click **Cancel**. Only tasks in the `Running` state can be canceled.
- **Task execution rules**: Tasks are executed in parallel. Efficiency depends on the global coroutine count configured in the system. Failure of one object class does not stop the task, but the task itself is marked as `Failed`.
- **Index replacement mechanism**: After a task succeeds, the system replaces the bound indexes of the object classes in one step. The replacement does not interrupt business queries. If the task fails, the original indexes are kept and current business usage is not affected.

## FAQ

Q1: **The system says "Creation failed" when I try to create an index build task. How do I fix it?**  
A1: Check the following:

- The selected object class is not bound to a view: go to **BKN > BKN > Object Class Management**, find the target object class, click **Bind View**, and configure the corresponding view.
- The selected object class already has a running task: go to **Task Management**, filter tasks in the `Running` state, confirm whether the target object class is included, and then wait for completion or click **Cancel** before creating the task again.

Q2: **If a task fails, what happens to the task itself and the existing indexes?**  
A2: Handle it as follows:

- Failure of one object class does not affect other tasks, but the task is marked as `Failed`.
- When a task fails, the original bound index of the object class is not replaced. Check the cause, such as abnormal view data or insufficient permissions, fix it, and create the task again.

Q3: **Can I modify an existing index build task, such as changing the task name or adjusting object classes?**  
A3: Not in the current version. Use this approach instead:

- Delete the original task. Only non-running tasks can be deleted.
- Recreate the task by following the **Create an Index Build Task** steps.

Q4: **What impact does a successful task have on later queries?**  
A4: The impact is as follows:

- After success, the system replaces the original bound index with the new one, and later queries against that object class automatically use the new index, improving both efficiency and semantic recall.
- The replacement runs in the background and does not interrupt ongoing queries, so business continuity is preserved.
